Developing Boost.Asio Extensions

Boost.Asio 라이브러리가 주로 네트워크 함수들을 지원하긴 하지만, 다른 비 동기 작업을 위한 I/O객체를 추가하는 것은 그리 어렵지 않다. 이 내용은 Boost.Asio 확장에 대한 내용이다. 물론 다음의 내용이, 의무적으로 따라야 하는 것은 아니지만, 또 다른 확장을 위한 출발점으로서, 실용적인 골격을 제공한다.

Boost.Asio에 새로운 비 동기 작업을 추가하기 위해서는, 3개의 클래스를 구현해야 한다.

- 새로운 I/O object 를 나타내는, boost::asio::basic_io_object 을 상속받은 것. 새로운 확장을 사용하는 개발자들은 이 I/O object 만을 사용하게 된다.

- boost::asio::io_service::service 를 상속받아, 서비스를 대표하는 클래스. I/O service로 등록돼서, I/O object 로부터 접근되어진다.  service 와 I/O object 의 구별이 중요한데,  I/O service 의 서비스 인스턴스 는 주어진 시간에 오직 하나만이 존재하지만, 다수의 I/O object 들이 하나의 서비스에 접근한다.

- 아무것도 상속받지 않는 서비스 구현을 위한 클래스. I/O service마다 오직 하나의 서비스 인스턴스만이 존재하므로, 서비스는 모든 I/O object를 위한 구현 클래스의 인스턴스를 각 각 생성한다.


단순히 골격을 제공하는 것 대신, 지금 개발하고자 하는 Boost.Asio 확장은 이미 사용 가능한 boost::asio::deadline_timer 객체와 유사한 모습일 것이다. 둘의 차이점은 타이머를 위한 기간이 생성자 대신, wait() 나 async_wait() 메서드 의 입력 인자로 전달된다는 것이다.

먼저 새로운 I/O object 를 나타내는 클래스부터 정의한다.
--------------------------------------------------------------------------------------------------------------------- 
// basic_timer.hpp
#include <boost/asio.hpp>
#include <cstddef>

template <typename Service>
class basic_timer : public boost::asio::basic_io_object<Service>
{
  public: explicit basic_timer(boost::asio::io_service &io_service)
        : boost::asio::basic_io_object<Service>(io_service)
    {
    }

    void wait(std::size_t seconds)
    {
        //포워딩
        return this->service.wait(this->implementation, seconds);
    }

    template <typename Handler>
    void async_wait(std::size_t seconds, Handler handler)
    {
        this->service.async_wait(this->implementation, seconds, handler);
    }
};

모든 I/O object 클래스는 보통 템플릿 클래스로 구현된다. 그리고 이 I/O object를 위해 특별히 개발된 서비스와 함께 인스턴스화 된다.
인스턴스 될 때마다 부모 클래스인 boost::asio::basic_io_object 에 의해서 I/O service로 자동적으로 등록된다.
이전에 이미 등록된 경우에는 재등록 되지 않기 때문에, I/O object에 의해 사용되는 서비스들은 I/O service별로 한번만 등록된다.

I/O object 안에서 해당 서비스는 'service' 참조를 통해서 접근할 수 있는데, 일반적으로 서비스로 메서드 호출을 포워딩 하기 위해서 필요하다.

서비스들이 모든 I/O object 의 데이터들을 저장할 필요가 있기 때문에, 'service'를 이용해서 모든 I/O object를 위한 인스턴스가 자동으로 생성된다. 이 역시 부모 클래스 boost::asio::basic_io_object의 도움으로 가능하다.
실제 서비스 구현(service implementation)은 모든 메서드 호출 시, 첫 번째 인자로 전달되는데 이를 통해 서비스는 어떤 I/O object가 호출했는지 알 수 있다. 서비스 구현은 'implementation' 속성으로 접근 가능하다.

일반적으로, I/O object 는 비교적 간단하다.
서비스의 설치 및 서비스 구현의 생성은 부모 클래스 boost::asio::basic_io_object을 통해 수행된다. 메서드 호출은 실제 I/O object의 서비스 구현을 인자로 해서, 해당 서비스로 단순히 포워딩 된다.

그다음 필요한것은 boost::asio::io_service::service 를 상속받는 서비스 클래스 이다.
---------------------------------------------------------------------------------------------------------------------
// basic_timer_service.hpp
#include <boost/asio.hpp>
#include <boost/thread.hpp>
#include <boost/bind.hpp>
#include <boost/scoped_ptr.hpp>
#include <boost/shared_ptr.hpp>
#include <boost/weak_ptr.hpp>
#include <boost/system/error_code.hpp>

template <typename TimerImplementation = timer_impl>
class basic_timer_service : public boost::asio::io_service::service
{
  public:
    static boost::asio::io_service::id id;

    explicit basic_timer_service(boost::asio::io_service &io_service)
        : boost::asio::io_service::service(io_service),
          async_work_(new boost::asio::io_service::work(async_io_service_)),
          async_thread_(boost::bind(&boost::asio::io_service::run, &async_io_service_))
    {
    }

    ~basic_timer_service()
    {
        async_work_.reset();
        async_io_service_.stop();
        async_thread_.join();
    }

    typedef boost::shared_ptr<TimerImplementation> implementation_type;

    void construct(implementation_type &impl)
    {
        impl.reset(new TimerImplementation());
    }

    void destroy(implementation_type &impl)
    {
        impl->destroy();
        impl.reset();
    }

    void wait(implementation_type &impl, std::size_t seconds)
    {
        boost::system::error_code ec;
        impl->wait(seconds, ec);
        boost::asio::detail::throw_error(ec);
    }

    template <typename Handler>
    class wait_operation
    {
      public:
        wait_operation(implementation_type &impl, boost::asio::io_service &io_service,
                       std::size_t seconds, Handler handler)
          : impl_(impl),
            io_service_(io_service),
            work_(io_service),
            seconds_(seconds),
            handler_(handler)
        {
        }

        void operator()() const
        {
            implementation_type impl = impl_.lock();
            if (impl)
            {
                boost::system::error_code ec;
                impl->wait(seconds_, ec);
                this->io_service_.post(boost::asio::detail::bind_handler(handler_, ec));
            }
            else
            {
                this->io_service_.post( boost::asio::detail::bind_handler(handler_,
                                        boost::asio::error::operation_aborted));
            }
        }
      private:
        boost::weak_ptr<TimerImplementation> impl_;
        boost::asio::io_service &io_service_;
        boost::asio::io_service::work work_;
        std::size_t seconds_;
        Handler handler_;
    };

    template <typename Handler>
    void async_wait(implementation_type &impl, std::size_t seconds, Handler handler)
    {
        this->async_io_service_.post( wait_operation<Handler>(impl,
         this->get_io_service(), seconds, handler));
    }

  private:
    void shutdown_service()
    {
    }
    boost::asio::io_service async_io_service_;
    boost::scoped_ptr<boost::asio::io_service::work> async_work_;
    boost::thread async_thread_;
};

template <typename TimerImplementation>
boost::asio::io_service::id basic_timer_service<TimerImplementation>::id;


Boost.Asio에 통합되기 위해서, 서비스는 몇 가지 요구 조건을 만족해야 한다.

- boost::asio::io_service::service을 상속받아야 한다.
  생성자는 I/O service 의 참조를 전달 받아야 한다.

- boost::asio::io_service::id 타입의 static public 속성인 id를 가져야 한다.
  I/O service내에서 서비스들은 이 속성으로 구분된다.

- implementation_type 타입의 인자를 갖는, public 메서드 construct() 와 destruct() 가 정의되어야 한다. implementation_type 은 보통 서비스 구현을 재정의 한 것이다. 위 예제에서처럼 boost::shared_ptr 객체가 서비스 구현을 인스턴스화 시키는데 쉽게 사용될 수 있다. construct(),destruct() 이들 메서드는 I/O object가 생성되고 소멸될 때 자동으로 호출되기 때문에, 서비스는 I/O object에 대해서 이 메서드를 이용해서 서비스 구현을 각각 생성하고 파괴할 수 있다.

- shutdown_service() 이름의 메서드가 반드시 정의되어야 한다(private이 가능).
  일상적인 Boost.Asio 확장에서 이것은 대부분 빈 메서드로 정의되는데, Boost.Asio와 좀더 강하게 통합된 서비스에 의해서만 별도로 정의되어 사용된다. 그렇지만 컴파일을 위해서 이 메서드를 반드시 정의해야만 한다.

해당 서비스로 메서드 호출을 포워딩 하기 위해, I/O object를 위한 메서드들이 정의 되어야 한다. 이 메서드들은  I/O object의 그것과 비슷하게 이름 지어진다(위 예제에서는 wait() 와 async_wait() 이다). wiat()과 같은 동기적 메서드는 서비스 구현에 단독으로 접근해서, 블럭킹 메서드 호출을 하고 있는 반면, async_wait()와 같은 비 동기 동작을 위해서는 쓰레드 안에서 블럭킹 메서드 호출을 하고 있다.

쓰레드의 도움으로 비 동기적 동작을 사용하는 것은, 보통 새로운 I/O service에 접근함으로써 이루어 진다.
위 예제에서는 boost::asio::io_service 타입의 async_io_service_ 라는 이름의 속성이 존재한다. 이 I/O service 의 생성자에서, boost::thread 타입의 async_thread_ 쓰레드를 시작하면서 쓰레드로 run() 메서드를 실행한다.

boost::scoped_ptr<boost::asio::io_service::work> 타입의 세번째 속성, async_work_ 는 run()이 즉시 리턴 하는 것을 막기 위해서 필요하다. 
http://www.boost.org/doc/libs/1_49_0/doc/html/boost_asio/reference/io_service__work.html

즉시 리턴 되는 것은, 생성시에 특별한 비동기 작업이 없기 때문에 일어 날수도 있다. 생성자 에서 boost::asio::io_service::work타입의 객체를 생성해서 I/O service에 바인드 시켜서 run()에서 즉시 리턴 되는 것을 방지한다.

서비스는 고유의 I/O service에 접근하지 않고도 구현될수 있다 - 하나의 쓰레드로 충분하다.
추가적인 쓰레드를 위해 새로운 I/O service를 사용하는 이유는 간단하다. 쓰레드끼리는 I/O service를 이용해서 쉽게 통신할 수 있기 때문이다. 예를 들어, async_wait()는 wait_operation 타입의 함수 객체를 생성해서, post() 메서드를 호출해서 내부의 I/O service로 전달한다. 내부 I/O service 의 run() 메서드를 실행하기 위해 사용됐던 쓰레드 내부에서, 이 함수 객체의 오더로드 된 operator()() 가 호출된다. post()는 다른 쓰레드에서 함수 객체를 실행하는 간단한 방법을 제공한다.

wait_operation 의 operator()()연산자에서는 wait()와 본질적으로 같은 작업을 수행한다.
즉, 서비스 구현의 blocking 메서드인 wait() 을 호출한다.
그런데 쓰레드가 operator()() 연산자를 실행할때 서비스 구현을 포함하는 I/O object가 파괴되었을 가능성도 있다. 만약 서비스 구현이 destruct()에서 파괴되었다면, operator()() 연산자는 더 이상 그것을 접근할 수가 없다.
이것은 weak pointer 를 이용해서 방지할 수 있다.
impl_ 은 lock() 호출 시, 서비스 구현이 존재하는 경우, 그것의 shared pointer를 리턴 해준다.
존재하지 않는다면 0을 리턴 한다(shared pointer 참조). 존재하지 않는다면 operator()()는 서비스 구현에 접근하지 않고, boost::asio::error::operation_aborted 에러 값으로 핸들러 를 호출한다.

마지막으로 서비스 구현 클래스이다.
--------------------------------------------------------------------------------------------------------------------- 
// timer_impl.hpp
#include <boost/system/error_code.hpp>
#include <cstddef>
#include <windows.h>

class timer_impl
{
  public: timer_impl() : handle_(CreateEvent(NULL, FALSE, FALSE, NULL))
    {
    }

    ~timer_impl()
    {
        CloseHandle(handle_);
    }

    void destroy()
    {
        SetEvent(handle_);
    }

    void wait(std::size_t seconds, boost::system::error_code &ec)
    {
        DWORD res = WaitForSingleObject(handle_, seconds * 1000);
        if (res == WAIT_OBJECT_0)
            ec = boost::asio::error::operation_aborted;
        else
            ec = boost::system::error_code();
    }

private:
    HANDLE handle_;
};


서비스 구현인 timer_impl 은 Windows API를 이용하고 있고, 몇 초 정도 기다리기 위해 호출될 wait() 메서드를 가지고 있다.
destroy()는 대기 동작을 취소하기 위한 것인데, 비 동기적인 동작을 위한 쓰레드 내부에서 wait()이 호출되기 때문에, 반드시 제공되어야 한다. 만약, 서비스 구현을 포함하는 I/O object가 파괴된 경우, 블럭된 wait() 호출은 destroy()를 이용해서 가능한 빨리 취소되어야 한다.

Boost.Asio 확장은 아래처럼 사용 되어진다.

#include <boost/asio.hpp>
#include <iostream>
#include "basic_timer.hpp"
#include "timer_impl.hpp"
#include "basic_timer_service.hpp"

void wait_handler(const boost::system::error_code &ec)
{
    std::cout << "5 s." << std::endl;
}

typedef basic_timer<basic_timer_service<> > timer;

int main()
{
    boost::asio::io_service io_service;
    timer t(io_service);
    t.async_wait(5, wait_handler);
    io_service.run();
}

새로운 확장은 boost::asio::deadline_timer 와 비슷하게 동작한다. 물론 이미 정의된 boost::asio::deadline_timer를 사용하는 것이 더 실용적일 것이다. 이 예제는 단지 새로운 비 동기 동작을 위해 Boost.Asio가 어떻게 확장가능한지 보여주기 위한 목적일 뿐이다.
Directory Monitor 는 I/O object가 디렉터리들을 모니터링 하는 Boost.Asio 확장의 실용적인 예제이다. 모니터링 되는 디렉터리 내에 파일이 생성, 변경,삭제될 때 핸들러가 호출된다(Windows와 Linux지원-Kernel version 2.6.13 or higher).
