실행중에 실패할수 있는 모든 함수들은, 자신을 호출한 쪽과 적절하게 정보를 주고 받을 방법이 필요하다.
C++ 에서는 리턴값이나 예외를 던짐으로서 이것을 해결한다. 발생한 에러가 아주 특별한 경우가 아니라면 
리턴값이 사용된다. 호출한 쪽에서는 리턴값을 확인해서 적함한 조치를 취해야 한다.
예외는 정상적인 경우에서는 나타날수 없는 특별한 조건임을 알리기 위해서 사용된다.
좋은 예로, new로 동적 메모리 할당이 실패할때 주어지는 std::bad_alloc 예외가 있다.
메모리 할당은 대부분 문제없이 이루어지기 때문에,  항상 리턴값을 확인한다는것은 매우 부담스러운 일이다.
이 장에서는 개발자들이 에러 처리를 도와주는 2개의 boost 라이브러리를 소개한다.
Boost.System은 운영체계에 의존적인 에러코드를 운영체계에 독립적인 것으로 변환시켜준다. 
이 덕분에 특정 운영체계에 기반한 값을 리턴하는 함수도 플랫폼 독립적으로 설계 가능하다.
Boost.Exception은 catch handler 에서 사용 가능한 예외에, 추가적인 정보를 덧붙일수 있게 해준다.

Boost.System



에러를 식별하기 위해 Boost.System 은 4개의 클래스로 구성된다.

boost::system::error_code

가장 기본적인 클래스로서, 운영체계의 특정한 에러를 나타낸다.
운영체계들은 보통 숫자로 에러를 표시하므로, boost::system::error_code 에서는
int유형으로 에러값을 저장한다. 다음은 Boost.Asio 사용시, 이 클래스를 사용하는 예제이다.

#include <boost/system/error_code.hpp>
#include <boost/asio.hpp>
#include <iostream>
#include <string>

int main()
{
    boost::system::error_code ec;
    std::string hostname = boost::asio::ip::host_name(ec);
    std::cout << ec.value() << std::endl;
}

Boost.Asio는 컴퓨터 이름을 리턴하는 boost::asio::ip::host_name() 자유함수를
제공한다. 이때 boost::system::error_code 타입의 객체가 인자로 전달될 수 있다.
만약 함수호출이 실패하는 경우, 이 객체에 상응하는 에러정보가 담겨진다.
에러코드가 상관없다면, 인자없이 boost::asio::ip::host_name()를 호출하면 된다.

Boost 1.36.0 버전에서, boost::asio::ip::host_name()는 잘못 동작했었다.
성공적으로 컴퓨터 이름을 리턴했어도 에러코드를 리턴했기 때문이다.
(Boost 1.36.0에서는 boost::asio::ip::host_name()를 잘못 구현한 이유로,
Windows XP, Visual Studio 2008에서 컴파일된 위 코드는
error code 14 (not enough storage available to complete the operation) 를
계속 발생시켰다) 이 문제는 Boost 1.37.0  에서 해결되었다.

에러는 숫자값에 지나지 않기 때문에, value() 를 이용해서 출력할수 있다.
에러코드 0 은 에러가 발생하지 않았음을 나타내지만, 다른 코드값은 운영체계에
의존적이므로 직접 의미를 찾아봐야 한다.

boost::system::category

value() 이외에, boost::system::error_code 는 category() 메서드를 제공한다.
이 메서드는 boost::system::category를 리턴한다.
에러코드는 간단한 숫자값이므로, 운영체계 제작사들은(예를 들면 MS)
시스템 에러코드값을 유일한 값으로 만들수 있겠지만, 어플리케이션 개발자가
모든 어플리케이션에서 유일한 에러코드를 만들어 내기란 불가능하다.
아마도 그건 모든 소프트웨어 개발자들의 에러코드를 저장시켜야 하는,
터무니 없는 중앙식 데이터베이스를 요구하게 될것이다.
이러한 이유로, 에러 카테고리가 존재한다.
모든 boost::system::error_code 유형의 에러코드들은 category()메서드 호출로
얻을 수 있는 카테고리에 항상 소속 된다.
운영체계 에러들은 이미 정의된 boost::system::system_category 객체에 의해서 표현된다.
category()를 호출함으로서, 이미 정의된 boost::system::system_category 객체의
참조가 리턴된다. 이를 이용하면 카테고리에 대한 특정 정보를 얻을수 있다.
예를 들어 name() 메서드를 호출해서 system category 경우에는 "system" 값을 얻을수 있다.

#include <boost/system/error_code.hpp>
#include <boost/asio.hpp>
#include <iostream>
#include <string>

int main()
{
    boost::system::error_code ec;
    std::string hostname = boost::asio::ip::host_name(ec);
    std::cout << ec.value() << std::endl;
    std::cout << ec.category().name() << std::endl;
}

출력:
0
system

에러는 에러코드와 에러 카테고리로 유일하게 구분된다.
이제 에러 코드들은  카테고리내에서만 유일하면 되기 때문에, 개발자들은
어플리케이션에서 사용할 특정 에러코드를 정의하고 싶을때마다 새로운
카테고리를 생성해야 한다. 이를 통해서 다른 개발자들의 에러코드와 충돌없는
임의의 에러코드를 사용할수 있다.

#include <boost/system/error_code.hpp>
#include <iostream>
#include <string>

class application_category : public boost::system::error_category
{
public:
    const char *name() const { return "application"; }

    //ev 인자를 이용해서 좀더 상세한 메시지 구성도 가능할것임
    std::string message(int ev) const { return "error message"; }
};

application_category cat;

int main()
{
    boost::system::error_code ec(14, cat);
    std::cout << ec.value() << std::endl;
    std::cout << ec.category().name() << std::endl;
}

출력:
14
application

새로운 카테고리 클래스를 정의하기 위해서는 boost::system::error_category를
상속해서, 순수 가상함수인 name() 와 message()를 재정의 해야 한다.
name() 가 에러 카테고리의 이름을 리턴하는 반면, message()는 특정 에러코드에 대한
에러 설명을 얻기 위해 사용된다. 예제에는 보여지지 않았지만 
대개 에러코드에 대한 설명을 리턴하기 위해서 ev인자가 평가된다.

새로 만든 에러 카테고리는 에러코드를 초기화 시킬때 사용가능하다.
예제에서는 에러코드 ec를 정의할때 새로만든 카테고리 application_category를 사용했다.
그러므로 이제 에러코드 14는 더이상 시스템 에러가 아니다.
그것의 의미는 새로운 에러 카테고리에, 개발자에 의해 지정되었다.

default_error_condition()

boost::system::error_code 는
boost::system::error_condition 타입의 객체를 리턴하는
default_error_condition() 메서드를 가지고 있다.

boost::system::error_condition 는 boost::system::error_code와
거의 동일한 인터페이스를 가지고 있다.
오직 하나의 차이점은 default_error_condition() 메서드는
boost::system::error_code 만이 가지고 있다는 점이다.

#include <boost/system/error_code.hpp>
#include <boost/asio.hpp>
#include <iostream>
#include <string>

int main()
{
    boost::system::error_code ec;
    std::string hostname = boost::asio::ip::host_name(ec);
    boost::system::error_condition ecnd = ec.default_error_condition();
    std::cout << ecnd.value() << std::endl;
    std::cout << ecnd.category().name() << std::endl;
}

출력:
0
generic

위 예제에서 boost::system::error_condition는 boost::system::error_code와
거의 같게 사용된다. value()와 category() 메서드가 동일하게 호출될수 있다.
이같이 비슷한 클래스를 2개나 만든 이유는 간단하다.
boost::system::error_code가 플랫폼 의존적인 에러코드를 위해 사용되는 반면,
boost::system::error_condition 는 플랫폼에 독립된 에러코드에 접근하기 위해 사용된다.

-----------------------------------------------------------------------------------------------------------------------
원래 글에서는 이 예제가 12를 리턴한다고 했지만, 이것은 Boost 1.36.0를 사용했을 경우고,
지금 테스트해보면 0(에러없음)을 리턴한다. 
저자가 테스트했던 당시 상황에서는, 플랫폼 의존적인 에러코드 14가
플랫폼에 독립적인 코드 12로 변환되었음을 보여주려고 한것임.
-----------------------------------------------------------------------------------------------------------------------

즉, 운영체계에 따라 에러코드값은 14가 될수도 있고 25가 될수도 있지만,
boost::system::error_condition 를 이용하면 언제나 같은 코드값을 받을수 있다.

boost::system::system_error

마지막으로 Boost.System가 제공하는 클래스는 boost::system::system_error 이다.
이클래스는 std::runtime_error 를 상속 받았다.
이것은 예외안에서 boost::system::error_code 타입의 코드를 전달시키기 위해 사용된다.

#include <boost/asio.hpp>
#include <boost/system/system_error.hpp>
#include <iostream>

int main()
{
    try
    {
        std::cout << boost::asio::ip::host_name() << std::endl;
    }
    catch (boost::system::system_error &e)
    {
        boost::system::error_code ec = e.code();
        std::cerr << ec.value() << std::endl;
        std::cerr << ec.category().name() << std::endl;
    }
}

위 예제에서, boost::asio::ip::host_name()는 2가지 버전이 존재하는데,
하나는 boost::system::error_code 인자를 받는것과 인자가 없는것이다.
후자의 경우에는 에러발생시 boost::system::system_error 타입의 예외를 던진다.
이경우 boost::system::error_code 얻기 위해 사용될 수 있다.
